Summary

  • Classification Ways

    • White Box (Internal known) and Black Box (Internal not known)

    • Experience Based Testing - Error Guessing, Exploratory and Checklist based

SDLC Software Development Life Cycle

  • Planning - What we want
    • Identification of the system development
    • Feasibility assessment
    • Creation of project plan
  • Defining - How we get what we want
    • Gathering Business Requirements
    • Creating Process Diagrams
    • Performing a detailed analysis
    • Designing of IT infrastructure
  • Building - Create what we want
    • Development of IT infra
    • Development of DB and code
  • Testing - Did we get what we want?
    • Writing test cases
    • Execution of test cases
  • Deploy - Lets start using what we got
  • Maintain - Lets get this closer to what we want

Smoke, Sanity and Regression Testing

https://www.udemy.com/course/software-testing-masterclass-from-novice-to-expert/learn/lecture/21201426#overview ::i A nice sanity, smoke and regression testing diagram here - https://www.udemy.com/course/software-testing-masterclass-from-novice-to-expert/learn/lecture/21201426#overview

  • Initial Build -> Smoke Testing
  • Stable Build --> Sanity Testing
  • If both above test pass --> System/ Regression/ Verification Testing

Smoke Testing

Performed at the initial development stage to ensure core functionalities are working fine. Not intended to be deep, but to verify core or main functionalities

When developers provide a fresh build to the QA team. When new module added to existing functionality.

Sanity Testing OR Surface Level Testing

Similar to Smoke, but here it happens after stable build gets a minor change or new module or functionality

Regression Testing

Check if even after all changes made, code still works. Its like ensuring enhancements or bug fixes

Here new tests are not created, but previously created tests are re-executed.

In short, after sanity testing, all impacted features are to be tested again. This is regression testing. Now you can see where the term regression comes from.

Runs after successful deployments.

Most commonly discussed. For each change, developer tests at his level, then ask senior/ team to run regression test to ensure.

Risk based testing

Theory - https://www.udemy.com/course/software-testing-masterclass-from-novice-to-expert/learn/lecture/14050275#overview Practice - https://www.udemy.com/course/software-testing-masterclass-from-novice-to-expert/learn/lecture/21255166#overview

Each problem has probability of happening.

Helps prioritize testing to obtain higher level of quality Risk is problem that might happen. More likely to happen, then riskier it is.

Use when other methods demand more time or resources than what can be afforded.

Risk = Impact x Likelihood

Over the period, technology will reveal pattern of characteristic problems

Static and Dynamic Testing

https://www.udemy.com/course/software-testing-masterclass-from-novice-to-expert/learn/lecture/14050279#overview

Dynamic Testing

Dynamic testing requires execution of code being tested

When: - Implementation - Acceptance Testing

Static Testing

Applied in the early stages of SDLC. Static testing is manual examination of test case or code reviews.

Asking for feedback on some test, its static testing.

When: - Requirement analysis and specification - Top-level design - Detailed Design

Example of what can be examined: - Business requirements - Test Cases - Code - User Guides - Contracts, project plans, schedules and budgets
